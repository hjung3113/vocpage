## 1. 미정의(Under-specified) 항목

### 🔴 Critical — 구현 시작 전 반드시 결정 필요

#### U-01. AD/SSO 인증 상세 미정의
- **위치**: §2.1, §6.1, §8.15
- **문제**: `validateADSession` 미들웨어가 "AD 인증 토큰 검증"이라고만 기재. LDAP bind / Kerberos / OAuth2 / 사내 JWT 중 어느 방식인지 불명확. 세션 만료 처리(리다이렉트? 토큰 갱신?), 쿠키 방식(httpOnly? SameSite?) 모두 미정의.
- **영향**: BE 미들웨어 설계 전체, FE 인증 흐름, 개발환경 mock bypass 전략 모두 차단
- **요청**: 사내 AD 연동 방식 확인 후 §6.1에 구체 스펙 추가

#### U-02. 공지사항 / FAQ DB 스키마 누락
- **위치**: §10.3, §10.4 — §4(데이터베이스 설계)에 해당 테이블 없음
- **문제**: `notices`, `faqs`, `faq_categories` 테이블이 §10에서 상세 기능이 명시됐지만 §4에 스키마가 없음. 컬럼, 인덱스, 제약 조건 미정의.
- **영향**: BE API 구현, DB 마이그레이션 파일 작성 차단
- **요청**: §4에 두 테이블 스키마 추가 (필드: §10.3.1, §10.4.1 기반)

#### U-03. FAQ 카테고리 저장 방식 미결정
- **위치**: §10.4.1
- **문제**: `카테고리 | enum (사용법/오류/정책/기타 등)`으로 정의했으나, 동시에 "Admin이 카테고리 목록 관리"라고 명시. DB enum 타입은 동적 추가 불가 → 별도 `faq_categories` 테이블이 필요한데 스키마 부재.
- **영향**: U-02와 동일 + ORM/마이그레이션 전략에 직접 영향
- **요청**: enum 고정 vs 별도 테이블 중 하나로 결정

#### U-04. 대시보드 API 엔드포인트 전무
- **위치**: §11
- **문제**: 위젯 9종의 데이터 조회 API가 전혀 정의되지 않음. 히트맵 드릴다운, SLA 집계, 담당자별 처리 현황 등 복잡한 집계 쿼리 설계 없음. `staleTime 5분`만 명시.
- **영향**: BE 집계 API 설계, FE 쿼리 키 설계, 성능 테스트 계획 모두 차단
- **요청**: §11에 각 위젯별 API 엔드포인트 + 요청/응답 스키마 추가

### 🟠 High — 구현 초기 단계에 결정 필요

#### U-05. Sub-task 순번(N) 범위 및 재사용 규칙 미정의
- **위치**: §8.7, §9.2
- **문제**: Sub-task ID `{parent-code}-{N}` — N의 패딩(1자리? 2자리?), 최대값, 삭제 후 N 재사용 여부 미정의. 부모 VOC issue_code는 재사용 금지인데(§8.1) Sub-task는 명시 없음.
- **영향**: Sub-task 생성 API, 목록 정렬 로직

#### U-06. 댓글 이미지 저장 위치 미정의
- **위치**: §8.5, §8.12
- **문제**: VOC 첨부파일은 `/uploads/{voc_id}/`에 저장. 그런데 댓글 본문(Toast UI Editor 인라인 이미지)의 저장 경로 미정의. VOC 첨부파일 경로와 동일 폴더? 별도 경로?
- **영향**: 파일 업로드 미들웨어 설계, storage_path 구조

#### U-07. 삭제된 VOC의 첨부파일 접근 정책 미정의
- **위치**: §8.9, §8.12
- **문제**: Soft Delete 후 `/files/:id`로 파일 접근 시 허용 여부 미정의. Admin이 `?includeDeleted=true`로 VOC 복원 전 파일 미리 확인 가능한지?
- **영향**: 파일 서빙 미들웨어 권한 로직

#### U-08. issue_code 5자리 확장 시 기존 코드 혼재 처리
- **위치**: §8.1
- **문제**: 순번 9999 초과 시 5자리로 자동 확장(`00001`). 그러나 4자리 코드와 혼재 시 사전식 정렬이 깨짐 (`ANALYSIS-2025-9999` > `ANALYSIS-2025-10000` 문자열 비교). `sequence_no` 컬럼이 별도이므로 정렬은 가능하지만 `issue_code` 표시 일관성 문제.
- **영향**: issue_code 표시 UI, 정렬 쿼리

#### U-09. 검색 하이라이팅 대상 불명확
- **위치**: §9.6
- **문제**: "VOC 목록의 제목·태그에서 매칭 텍스트 강조"라고 했는데, 태그는 자동 부여된 태그명 전체가 일치해야 하이라이팅? 부분 문자열 일치? 본문(body)은 목록에 미표시인데 본문 검색 결과를 어떻게 시각화?
- **영향**: FE 하이라이팅 컴포넌트 설계

#### U-10. 알림 폴링 에러 처리 미정의
- **위치**: §8.14
- **문제**: 30초 폴링 중 네트워크 오류 / 401 응답 시 재시도 전략, 백오프 정책 미정의.
- **영향**: FE 폴링 훅 구현

### 🟡 Medium — 구현 중반부에 결정 필요

#### U-11. VOC 전체화면 URL 상태 "선택적" 모호
- **위치**: §9.9
- **문제**: "URL 파라미터(`?view=full`) 또는 sessionStorage로 유지 (새로고침 시 복원 선택적)"에서 "선택적"이 구현자 재량인지 명확하지 않음.
- **영향**: FE 라우팅 설계

#### U-12. Slug 고유성 제약 미명시
- **위치**: §4 (`systems`, `menus`, `voc_types`)
- **문제**: 세 테이블 모두 `slug` 컬럼이 있지만 UNIQUE 제약 범위 미정의. `systems.slug`은 전역 고유? `menus.slug`은 시스템 내 고유?
- **영향**: DB 마이그레이션 제약 설계

#### U-13. 대시보드 잠금(locked_fields) Granularity 불명확
- **위치**: §11.5, §4 `dashboard_settings`
- **문제**: `locked_fields(jsonb)`에 어떤 키가 들어가는지 미정의. 위젯 단위 잠금인지, 위젯 내 특정 속성(크기/위치/가시성) 단위인지 불명확.
- **영향**: JSONB 스키마 설계, Admin 편집 UI

#### U-14. Sub-task 첨부파일 카운트 기준 미정의
- **위치**: §8.5, §8.7
- **문제**: "최대 5개/VOC (본문 기준)"에서 Sub-task도 독립적으로 5개 허용인지, 부모 VOC와 합산인지 미정의.
- **영향**: 파일 업로드 API 검증 로직

#### U-15. 상태 관리 방식(React Context vs Redux) 미결정

- **영향**: FE 아키텍처 (구현 시작 전까지만 결정하면 됨)

## 2. 모순(Contradiction) 항목

### 🔴 Critical

#### C-01. Sub-task 최대 단계 표현 충돌
- **§2.3**: "하나의 이슈 아래 여러 하위 작업(Sub-task)을 생성 — **최대 2단계**"
- **§8.7**: "1레벨만 허용: `vocs` → Sub-task. Sub-task의 Sub-task 생성 불가"
- **§9.2**: "부모 VOC가 이미 Sub-task인 경우 하단 폼 비활성화 (**최대 2단계 제한**)"
- **해석**: §8.7이 정확한 스펙. §2.3의 "2단계"는 "Root VOC(1단계) + Sub-task(2단계)"를 가리키는 것으로 읽혀야 하나, 일반적으로 "2단계 Sub-task"로 오해 가능.
- **조치**: §2.3 표현을 "최대 1레벨 Sub-task (root → sub 단일 계층)"으로 수정

#### C-02. FAQ 카테고리 enum vs 동적 관리 직접 충돌
- **§10.4.1**: `카테고리 | enum (사용법/오류/정책/기타 등)`
- **§10.4.1**: "Admin이 카테고리 목록 관리"
- **충돌**: DB enum은 ALTER TABLE 없이 동적 추가 불가. 동적 관리를 원한다면 별도 테이블 필수.
- **조치**: `faq_categories` 테이블로 변경, §4에 스키마 추가, §10.4.1에서 "enum" 표현 제거

#### C-03. 태그 수동 편집 권한이 권한 테이블에 누락
- **§8.8 태그 섹션**: "Admin/Manager는 오탐 정정을 위해 개별 VOC의 태그를 수동으로 추가/삭제 가능"
- **§8.3 권한 모델 테이블**: 태그 수동 편집 행이 없음
- **조치**: §8.3 권한 테이블에 "태그 수동 편집" 행 추가 (User ❌ / Manager ✅ / Admin ✅)

---

### 🟠 High

#### C-04. 알림 보관 정책 중복 기준
- **§8.6**: "최근 50건 표시" AND "30일 이후 자동 읽음 처리"
- **충돌**: 50건 제한과 30일 제한이 독립적으로 적용되면 어느 쪽이 우선인지 불명확. 예) 30일 미만인데 51번째 알림은? 30일 이후 읽음 처리만 하고 삭제는 안 하는가?
- **조치**: 보관 정책을 "30일 이후 자동 삭제 OR 51번째부터 오래된 순으로 삭제 — 둘 중 먼저 도달한 기준 적용"으로 명확화

#### C-05. 댓글 삭제만 Hard Delete — 감사 추적 불일치
- **§8.13**: 댓글 삭제 = Hard Delete (본인/Admin)
- **§8.9**: VOC 삭제 = Soft Delete
- **§10.3.4, §10.4.3**: 공지/FAQ 삭제 = Soft Delete
- **문제**: 댓글에도 민감 정보가 포함될 수 있고, 관리자 감사 목적으로 보존이 필요할 수 있음. 의도적 결정이라면 이유를 ADR로 남겨야 함.
- **조치**: Hard Delete 유지 여부 명시적 결정 후 §8.13에 "의도적 Hard Delete — 이유: ___" 주석 추가

#### C-06. 공지 복원 권한 — Manager 범위 불명확
- **§10.3.4**: "Admin은 관리 목록에서 복원 가능"
- **§10.1**: 공지사항 "작성/관리 권한: Admin, Manager"
- **문제**: Manager가 삭제한 공지를 Manager도 복원할 수 있는지, Admin만 가능한지 불명확.
- **조치**: §10.3.4에 복원 권한 명시 (Admin 전용? Admin+Manager?)

---

### 🟡 Medium

#### C-07. 대시보드 Manager의 잠금 설정 권한
- **§11.5 편집 모드**: "항목별 🔒 잠금 (Admin만)"
- **§11.5 저장 방식**: "내 설정 저장: 로그인 사용자 개인 레이아웃 ... 잠기지 않은 항목만"
- **문제**: Manager도 레이아웃 편집 가능(§11.5 접근 권한)하지만, 잠금은 Admin 전용. Manager가 편집 모드에서 잠금 아이콘을 볼 수는 있지만 클릭 불가인지, 아예 미노출인지 UI 명세 없음.
- **조치**: 잠금 아이콘 Manager 뷰에서 disabled 표시 여부 명시

#### C-08. 사이드바 카운트 배지 — "전체 VOC" User 접근 불가와 충돌
- **§10.2.2**: "`전체 VOC` 배지: 항상 표시"
- **§8.3 권한 모델**: User는 "전체 VOC 조회 ❌"
- **문제**: User가 사이드바에서 전체 VOC 수를 배지로 볼 수 있다면 간접적으로 정보 노출. 또는 User 뷰에서 "전체 VOC" 탭 자체가 숨겨져야 하는지 명시 없음.
- **조치**: User 뷰에서 "전체 VOC" 탭 숨김 여부 명시

## 3. 전문가 5인 관점 체크리스트

---

### Expert 1: Product Manager (PM) — 비즈니스 가치 & 범위

> *"이 시스템이 실제 팀 운영에 즉시 가치를 제공하는가?"*

#### PM-1. MVP 범위가 실제 운영에 충분한가
- **질문**: SLA 없음, 벌크 액션 없음, 마감일 없음, 이메일 알림 없음이 MVP에서 제외됨. 운영팀이 인앱 알림만으로 긴급 VOC를 놓치지 않을 수 있는가?
- **현재 상태**: ⚠️ 불명확
- **심각도**: High
- **체크**: [ ] 인앱 폴링 30초 + 99+ 배지로 긴급 알림 누락 없다는 운영팀 컨펌 필요
- **권고**: MVP 이후 1순위로 이메일 알림 추가 일정 수립. §7 확장성 로드맵에 우선순위 명시

#### PM-2. 권한 모델 3단계(User/Manager/Admin)가 실제 조직과 매핑되는가
- **질문**: 현실에서 "Manager"가 누구인가? 팀장? 서비스 담당자? Manager 역할 부여 주체(Admin)가 AD 그룹 기반인지 수동 지정인지 미정의.
- **현재 상태**: ❌ 미정의
- **심각도**: High
- **체크**: [ ] AD 그룹 → 역할 자동 매핑 여부 결정
- **체크**: [ ] 초기 Admin 계정 생성 방법(seed? 환경변수?) 명시

#### PM-3. 확장성 로드맵이 데이터 모델과 연결되어 있는가
- **질문**: §7에서 "AI 자동 분류", "데이터 분석 리포트"를 언급했지만 현재 데이터 모델에 LLM API 호출 이력, 분류 신뢰도 등을 저장할 컬럼이 없음. 나중에 추가하면 마이그레이션 비용 발생.
- **현재 상태**: ⚠️ 불명확
- **심각도**: Medium
- **체크**: [ ] `tag_rules` 테이블에 `confidence_threshold` 컬럼 등 AI 전환 대비 필드 여부 결정

### Expert 2: UX Designer — 사용성 & 인터랙션

> *"사용자가 시스템을 직관적으로 사용할 수 있는가?"*

#### UX-1. 복수 팝업 2-panel 모달의 인지 부하
- **위치**: §10.3.2
- **질문**: 여러 팝업 공지가 있을 때 좌(목록)+우(상세) 2-panel 모달을 처음 보는 사용자가 "오늘 하루 보지 않기"가 현재 선택 공지에만 적용되는지, 전체에 적용되는지 혼란 가능. 스펙에는 "전체 일괄 적용"으로 명시했으나 단일 공지에도 "오늘 하루 보지 않기"가 필요한가?
- **현재 상태**: ⚠️ 불명확
- **심각도**: Medium
- **체크**: [ ] 단일 팝업 시에도 동일 2-panel? 아니면 단일 팝업은 전통적 모달?
- **체크**: [ ] "오늘 하루 보지 않기" → 개별 공지 단위 vs 전체 세션 단위 결정

#### UX-2. 연계 드롭다운 접근성 및 빈 상태 처리

- **심각도**: High

- **심각도**: High

### Expert 5: Security Engineer — 보안 & 접근 제어

> *"시스템이 공격과 오용에 안전한가?"*

#### SEC-1. 파일 업로드 보안 완전성
- **위치**: §8.5
- **문제**: "MIME 스니핑 + 확장자 일치 검증. 실행 파일 헤더 차단"이라고 명시. 그러나:
  - SVG 파일은 허용 목록에 없지만 XSS 위험 (명시적 차단 불명확)
  - GIF는 GIF bomb(재귀 압축) 가능 — 파일 크기 10MB 제한이 디코딩 후 크기인지 원본인지?
  - 서빙 경로(`/files/:id`)에서 Content-Disposition: attachment 강제 여부 미정의 (브라우저 직접 실행 방지)
- **현재 상태**: ⚠️ 불명확
- **심각도**: Critical
- **체크**: [ ] SVG 명시적 차단 추가
- **체크**: [ ] `Content-Disposition: attachment` 강제 명시
- **체크**: [ ] GIF 파일 크기 검증 기준(원본 바이트) 명시

#### SEC-2. 개발환경 AD bypass 취약점 가능성
- **위치**: §6.1, next-session-tasks §6-6
- **문제**: "NODE_ENV=development 시 mock user 주입"은 흔한 패턴이나, NODE_ENV 환경변수가 프로덕션에서 실수로 development로 설정될 경우 인증 우회. 이를 방지할 보조 안전장치 미정의.
- **현재 상태**: ❌ 미정의
- **심각도**: Critical
- **체크**: [ ] bypass 조건을 `NODE_ENV=development AND ALLOW_MOCK_AUTH=true` 이중 플래그로 강화
- **체크**: [ ] 프로덕션 배포 시 ALLOW_MOCK_AUTH 미설정 확인 체크리스트 추가

#### SEC-3. 권한 상승 방지 — 마지막 Admin 외 추가 케이스
- **위치**: §8.3, §9.4.4
- **현재 명시**: 마지막 Admin 강등 불가 + 본인 권한 변경 불가
- **미정의 케이스**:
  - Admin A가 Admin B를 Manager로 강등하면서 동시에 Admin C를 Manager로 강등하는 트랜잭션?
  - `is_active = false`인 Admin이 있을 때 활성 Admin이 1명뿐인 경우 강등 가능?
- **현재 상태**: ⚠️ 불명확
- **심각도**: High
- **체크**: [ ] "마지막 Admin" 판정 기준을 "is_active = true AND role = admin 인 계정이 2명 이상"으로 명시
- **체크**: [ ] 권한 변경 API에 트랜잭션 원자성 + 사후 Admin 수 검증 명시
