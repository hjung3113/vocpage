## U. 사용자 메타 관점

> **"언제나 누가 보든 해당 요구사항 문서를 보면 동일한 시스템이 만들어질 정도로 구체적인가?"**

### M1. 독자 일관성 점수 — 5/10

10명의 개발자가 이 문서를 독립적으로 읽었을 때, **동일한 시스템**이 나오지 않을 가능성이 높은 항목:

| 항목 | 개발자마다 달라질 것 | 심각도 |
|---|---|---|
| AD 인증 구현 방식 | LDAP, OIDC, 사내 JWT → 완전히 다른 미들웨어 | 🔴 Critical |
| 전체화면 전환 복원 | "선택적" → 구현하는 사람 vs 건너뛰는 사람 | 🟠 High |
| Sub-task 레벨 | §2.3 "2단계" vs §8.7 "1레벨" → 2레벨 허용으로 잘못 구현 가능 | 🔴 Critical |
| FAQ 카테고리 저장 | enum vs 별도 테이블 → DB 구조 완전히 달라짐 (이미 수정됨) | ✅ 해결됨 |
| 대시보드 API 설계 | endpoint 없음 → 각자 다른 API 구조 | 🔴 Critical |
| 댓글 이미지 저장 경로 | 경로 기술 있으나 `attachments` 행 생성 여부 | ✅ 해결됨 |
| 알림 보관 정책 우선순위 | 50건 vs 30일 중 어느 쪽? | ✅ 해결됨 |
| 검색 하이라이팅 범위 | 태그 부분 일치 여부 | ✅ 해결됨 |

**정량 분석:**
- 독립 구현 시 동일 결과 예상 항목: 약 12/20개 주요 기능
- 구현자마다 달라질 고위험 미결정: 3건 (AD인증, Sub-task표현, 대시보드 API)
- 의미있는 모호성(사소): 5건

**결론:**
> 기능 스펙(권한, 상태 전환, 파일 정책)은 잘 정의되어 있어 핵심 도메인 로직은 재현 가능하다.  
> 그러나 **AD 인증 방식, 대시보드 API, §2.3 Sub-task 표현** 이 3건이 해결되지 않으면 팀원마다 다른 시스템이 나온다.  
> 운영·배포·테스트 전략은 이 문서에서 거의 읽어낼 수 없어, 운영 품질의 일관성은 보장하기 어렵다.
