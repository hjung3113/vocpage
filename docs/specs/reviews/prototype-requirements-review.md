## [C] 요구사항 문서의 모순·모호점

### C1. 첨부파일 용량 제한 — **10 MB vs 20 MB 모순**
- §8.5 "최대 크기 **10 MB**/파일"
- `prototype.html` 공지 샘플(`NOTICES[1]`): "첨부파일 용량 제한 변경 안내 (**10MB → 20MB**)"
- `prototype.html` FAQ 샘플(`FAQS[2]`): "지원 형식: … (최대 **20MB** / 개, 5개 / VOC)"
- 문제: 공지/FAQ는 데모 콘텐츠지만 사용자에게 노출되는 정책 텍스트 → 스펙과 일치시켜야 함.
- **제안:** 스펙을 단일 source of truth 로 두고, 프로토타입의 공지/FAQ 샘플 문구를 10MB 기준으로 수정하거나, 스펙을 20MB로 올릴 지 결정.

### C2. 사용자 역할(enum) — **요구사항 3종 vs 프로토타입 5종 불일치**
- §2.1·§8.3: `user / manager / admin` (3종)
- `ADMIN_USERS` / `ROLE_MAP`: `admin / manager / dev / viewer` (4종) — 그리고 일반 `user` 는 빠져 있음.
- **제안:** 두 갈래 중 선택
  - (a) 스펙을 3종으로 유지 → 프로토타입에서 `dev`·`viewer` 제거, `user` 추가
  - (b) 스펙을 확장 (`dev`·`viewer`를 정식 역할로 정의하고 권한 매트릭스 §8.3 갱신)

### C3. 서브태스크 생성 폼 필드 — 제목만 vs 제목+유형
- §9.2 "드로어 하단 폼: 제목 입력 → 저장/취소 버튼"
- §8.7 "Sub-task 는 독립적인 상태·담당자·Priority·**유형** 보유" + "Sub-task 의 유형은 부모와 독립적으로 선택 가능" (§8.8 유형)
- 프로토타입: 제목 + 유형 select 노출 (유형 독립 선택 허용).
- 모호점: §9.2 에서 "제목 입력" 이 최소 필드를 의미하는지, 단일 필드만 허용하는지 불분명.
- **제안:** §9.2 를 "제목 + (선택) 유형 → 저장/취소" 로 보완하고, 유형 미지정 시 기본값(예: 부모와 동일) 규정.

### C4. 공지 팝업 정렬 — "중요도 내림차순" 모호
- §10.3.2 "복수의 팝업 공지가 있을 경우 **중요도 내림차순**으로 순서 표시"
- "내림차순" 이 "가장 중요한 것이 위" 를 뜻하는지, 문자 코드 내림차순인지 불분명.
- 프로토타입은 `urgent(0) → important(1) → normal(2)` 오름차순 정렬이지만 시각적으론 "긴급이 맨 위" 라 스펙이 의도한 바와 맞는 것으로 추정.
- **제안:** "중요도 상위(urgent → important → normal) 순" 으로 문구 교체.

### C5. 공지사항 "삭제" — Soft Delete vs 완전 삭제 문구 충돌
- §10.3.4 "삭제: 완전 삭제 가능 (**Soft Delete — DB에는 남기고 목록에서 제거**)"
- 문제: "완전 삭제" 와 "Soft Delete" 는 상반된 개념. Soft Delete 이면 복원 가능해야 하는데 복원 동선 미명시.
- **제안:** "Soft Delete (목록 숨김, Admin 복원 동선 ○/×)" 로 명확화.

### C6. "전체 VOC 조회" 권한
- §8.3 "전체 VOC 조회: User ❌, Manager ✅, Admin ✅"
- §2.2 "제보자(= User) — 본인 VOC 실시간 상태 확인"
- OK (충돌 없음). 다만 §8.11 필터·검색이 User 에게도 작동해야 하는 범위가 "본인 VOC 내"로 좁혀지는지 미기술.
- **제안:** 필터 스코프 = 해당 사용자가 조회 가능한 범위로 자동 축소 — 라고 §8.11 에 한 줄 추가.

### C7. "미완료 Sub-task 있는 부모를 완료로 전환 시 경고 **강제 진행 가능**"
- §8.2 "경고 메시지 후 강제 진행 가능" — 강제 진행한 경우 Sub-task 상태는? (부모만 완료 / 자식도 cascade 완료 / 자식은 유지)
- **제안:** 자식 Sub-task 상태는 유지(관리자가 개별 처리) 로 고정하고 스펙 보강.

### C8. "시스템명 한글 2~16자" (§8.10) vs 실제 사례
- 샘플 "데이터 파이프라인" = 8자지만 공백 포함. "한글 2~16자" 에 공백/숫자/영어 혼용 허용 여부 불분명.
- **제안:** "최소 2자·최대 16자 (공백 포함, 한글/영문/숫자 허용)" 로 보강.

### C9. `issue_code` 에 쓰이는 "시스템 슬러그" — 한글 슬러그 URL-safe 가능?
- §4: `slug(URL-safe)` + §8.1 예시 `인증-2025-0001` (한글) → 서로 충돌.
- 한글은 URL 인코딩은 가능하지만 "URL-safe" 의 관용적 의미(ASCII)와 맞지 않음.
- **제안:** 스펙을 "URL 경로에 사용 가능한 식별자 (인코딩 허용)" 로 완화하거나, 슬러그는 영문, `issue_code` 는 한글 병기 방식(예: `ANALYSIS-2025-0001`)으로 분리.

### C10. VOC 목록 기본 page size
- §8.11 "기본 25개/페이지"
- §9.8 "화면 높이에 따라 동적 계산 (최소 10개)"
- 두 문장이 공존. §9.8 이 §8.11 를 **대체**하는지 **오버라이드**하는지 명시 없음.
- 프로토타입은 §9.8 를 따름 (고정 25 아님).
- **제안:** §8.11 의 "기본 25개" 문구를 §9.8 의 동적 계산 공식으로 일원화.

### C11. 태그 M:N (voc_tags) 정책과 자동 태깅만 허용 정책의 정합
- §4 `tags`, `voc_tags` 는 M:N.
- §8.8 "태그 — 자동 태깅 전용 — 수동 추가/삭제 불가".
- Admin/Manager 가 특정 VOC 의 태그를 오탐 정정(삭제/추가)할 수 있는 동선이 없음.
- **제안:** "Admin 은 개별 VOC 의 태그를 수동 정정 가능 (이력 기록)" 로 보완하거나, "엄격히 자동 태깅만" 을 재확인.
